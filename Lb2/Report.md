# Звіт до Лабораторних робіт №1 та №2 (QA Automation)

## Короткий звіт відповідності вимогам

| Код фрагмента | Опис того, яку саме вимогу (з ЛБ 1 чи ЛБ 2) він виконує |
| :--- | :--- |
| `[TestCase("-10", "5", "+", -5.0)]` | **ЛБ 1:** Використання параметризованих тестів (`[TestCase]`) для перевірки **класів еквівалентності** (від'ємні та додатні числа, різні типи вводу). |
| `[TestCaseSource(nameof(BoundaryValuesSource))]` та `"1.79769313...E+308"` | **ЛБ 1:** Перевірка **граничних значень** типу `double` (максимальне/мінімальне значення та робота з нескінченністю). Передача великих наборів даних через `TestCaseSource`.|
| `Assert.Throws<DivideByZeroException>(...)` | **ЛБ 1:** Перевірка виникнення відповідних **виключень** (Exceptions) при неприпустимих негативних сценаріях (ділення на нуль, невалідний текст). |
| `Is.EqualTo(ErrorMessages.DivideByZero)` | **ЛБ 1:** Уникнення **"хардкодингу"**. Повідомлення про помилки винесені в константи класу `ErrorMessages`, що дозволяє легко їх змінювати без переписування всіх тестів. |
| `public class FakeLogger : ILogger` | **ЛБ 1 та ЛБ 2:** Застосування патерну Mocking (створення **Fake** об'єкта) для **ізольованого тестування**. Тести калькулятора перевіряють логіку без виведення даних у реальну консоль чи файл. |
| `public Calculator(ILogger logger)` | **ЛБ 2:** Використання паттерну **Dependency Injection (DI)** (ін'єкція залежностей). Дозволяє легко підміняти реальний логер на фейковий під час тестування. |
| `[TestFixture]`, `[SetUp]` | **ЛБ 2:** Використання стандартної архітектури **NUnit** фреймворку. `SetUp` гарантує ініціалізацію "свіжих" екземплярів перед кожним окремим тестом. |
| Клас `StringProcessorTests` | **ЛБ 2:** Створення модульних тестів для ізольованого методу `IsPalindrome`. Використання специфічних стверджень `Assert.IsTrue` та `Assert.IsFalse`. |

## Відповіді на контрольні питання

### Контрольні питання ЛБ 1 (Особливості тест-кейсів)
1. **Що таке класи еквівалентності та для чого їх виділяють?**
   *Відповідь:* Це техніка чорноскринькового тестування. Всі можливі вхідні дані розбиваються на групи (класи), які система обробляє однаково. Замість перевірки 1000 різних додатних чисел, достатньо перевірити одне число з "позитивного класу еквівалентності", одне з "негативного" і одне з форматом, що містить помилку. Це суттєво зменшує кількість тестів при збереженні рівня покриття.
2. **Що таке аналіз граничних значень?**
   *Відповідь:* Практика показує, що більшість багів ховаються на межі між класами еквівалентності (наприклад, перехід від 0 до 1, або дуже великі числа, що межують з лімітом пам'яті). Тестування границь (менше межі, на межі, більше межі) гарантує стабільність.
3. **Чому в тестах та коді варто уникати розміщення "хардкод" рядків (англ. Hardcoding)?**
   *Відповідь:* Якщо повідомлення "Ділення на нуль неможливе." замінять на "Неможливо поділити на нуль.", доведеться вручну переписувати десятки тестів. Якщо значення тримати у змінних-константах (Constants / Enums), зміну тексту треба буде зробити лише в одному місці.

### Контрольні питання ЛБ 2 (Модульні тести)
1. **Яке головне призначення модульних (Unit) тестів?**
   *Відповідь:* Модульне тестування перевіряє найменші "модулі" коду (функції, класи, методи) абсолютно ізольовано від інших частин програми, бази даних та інтерфейсу, щоб швидко виявляти регресію та логічні помилки на етапі написання коду.
2. **Навіщо потрібні Mock/Fake/Stub об'єкти під час модульного тестування?**
   *Відповідь:* Для забезпечення *ізоляції*. Якщо наш калькулятор пише логи в базу даних, "справжній" логер намагатиметься підключитися. Ми інжектуємо (DI) `FakeLogger`, який просто збирає логи в список у оперативній пам'яті, дозволяючи швидко і безпечно перевірити систему.
3. **Функції атрибутів `[TestFixture]`, `[SetUp]` та `[Test]` в NUnit?**
   *Відповідь:* `[TestFixture]` вказує компілятору, що в цьому класі лежать тести. `[Test]` або `[TestCase]` маркують безпосередньо тестові методи, які повинен запустити Test Runner. `[SetUp]` позначає метод, який буде автоматично виконуватися *перед запуском кожного окремого* тесту, щоб гарантувати стабільне та чисте тестове середовище.
