# Звіт з Лабораторної роботи №4
**Тема:** Використання BDD підходу та фреймворку Cucumber

## 1. Використані технології та стек
Під час виконання лабораторної роботи був використаний наступний стек технологій:
- **Мова програмування:** JavaScript (середовище Node.js)
- **BDD Фреймворк:** Cucumber.js (`@cucumber/cucumber`) — для написання та читання специфікацій у форматі Gherkin (природною мовою).
- **Інструмент автоматизації браузера:** Selenium WebDriver (`selenium-webdriver`) — для імітації дій реального користувача системі (введення тексту, кліки).
- **Бібліотека перевірок (Assertions):** Chai (`chai`) — для строгої перевірки (звірки) очікуваних результатів із фактичними.
- **Генерація звітів:** Allure (`allure-cucumberjs` та `allure-commandline`) — інструмент для візуалізації результатів тестування у вигляді красивих та детальних HTML-сторінок зі скріншотами.

## 2. Як працює програма (Запуск та Allure)
Процес розбитий на дві окремі фази: тестування та генерація звіту.

### Етап 1: Запуск тестів (Команда `npm test`)
Вона звертається до Cucumber. Фреймворк знаходить файл специфікацій `features/atm.feature` та зчитує Gherkin-алгоритми. Паралельно він запускає код із файлу `steps.js`. 
За допомогою `selenium-webdriver` у фоновому (headless) режимі браузер Google Chrome відкриває нашу локальну веб-сторінку (`atm.html`), вводить значення балансу, суми зняття та натискає кнопку. Після цього через бібліотеку Chai звіряється, чи відповідає текст на екрані очікуваному.
Всі результати кожного дрібного кроку тесту зберігаються як "сирі" системні лог-файли (JSON/XML формату) в автоматично створену папку `allure-results`.

### Етап 2: Збірка звіту Allure (Команда `npm run report`)
Після того, як тести завершені та є "сирі" дані, ми запускаємо Allure. Ця команда зчитує всі результати з папки `allure-results`, аналізує їх і формує красивий HTML-сайт у папку `allure-report`. Також команда відразу підіймає локальний сервер і відкриває цей звіт у вашому браузері. Allure відображає графіки, тривалість виконання, а у випадку провалу тесту — прив'язує зроблені через `hooks.js` скріншоти.

## 3. Реалізація кроків (Step Definitions)

Нижче наведено мапінг регулярних виразів Gherkin на реальний JavaScript-код (взаємодію з браузером через `selenium-webdriver`), реалізований у файлі `steps.js`:

```javascript
Given('на моєму рахунку є {int} гривень', async function (balance) {
    const filePath = 'file:///' + path.resolve(__dirname, '../../atm.html').replace(/\\/g, '/');
    await this.driver.get(filePath);

    const balanceInput = await this.driver.findElement(By.id('balanceInput'));
    await balanceInput.clear();
    await balanceInput.sendKeys(balance.toString());
    
    const setBalanceBtn = await this.driver.findElement(By.id('setBalanceBtn'));
    await setBalanceBtn.click();
});
```
**Опис:** Цей крок виконує підготовчі дії (Preconditions). Він відкриває локальну веб-сторінку нашого макета банкомата (`atm.html`) у браузері. Далі він знаходить поле введення балансу (за його ідентифікатором `balanceInput`), очищує його та вводить початкову суму, передану з Gherkin-файлу як аргумент `{int}`. Наприкінці скрипт імітує клік на кнопку "Встановити баланс", записуючи дані в систему.

```javascript
When('я намагаюся зняти {int} гривень', async function (amount) {
    const amountInput = await this.driver.findElement(By.id('amountInput'));
    await amountInput.clear();
    await amountInput.sendKeys(amount.toString());
    
    const withdrawBtn = await this.driver.findElement(By.id('withdrawBtn'));
    await withdrawBtn.click();
});
```
**Опис:** Це ключова дія (Action) нашого тесту. Скрипт знаходить інпут із сумою для зняття (`amountInput`), вводить туди число (яке ми прагнемо зняти) і клікає на відповідну кнопку "Зняти готівку", ініціюючи бізнес-логіку самого додатка.

```javascript
Then('банкомат повинен видати {int} гривень', async function (expectedDispensed) {
    const dispensedEl = await this.driver.findElement(By.id('dispensedAmount'));
    const text = await dispensedEl.getText();
    expect(parseInt(text, 10)).to.equal(expectedDispensed);
});
```
**Опис:** Перша перевірка результату (Assertion). Скрипт зчитує текстове значення елемента на сторінці, у якому банкомат виводить реально видану суму. Далі він перетворює текст на ціле число та звіряє його з очікуваною сумою (за допомогою бібліотеки Chai: команда `expect(...).to.equal(...)`). Якщо цифри не збігаються (наприклад, ми очікуємо успіх 500, а система видала 0 через брак коштів) — тест миттєво "падає" з червоною помилкою.

```javascript
Then('мій залишок на рахунку повинен бути {int} гривень', async function (expectedBalance) {
    const balanceEl = await this.driver.findElement(By.id('currentBalance'));
    const text = await balanceEl.getText();
    expect(parseInt(text, 10)).to.equal(expectedBalance);
});
```
**Опис:** Друга перевірка результату. Скрипт зчитує актуальний залишок коштів на рахунку після виконаної операції (з елемента `currentBalance`) і порівнює з тим залишком, який ми передбачили в таблиці Прикладів нашого `atm.feature` файлу.

```javascript
Then('на екрані має з\'явитися повідомлення {string}', async function (expectedMessage) {
    const messageEl = await this.driver.findElement(By.id('message'));
    const text = await messageEl.getText();
    expect(text).to.equal(expectedMessage);
});
```
**Опис:** Фінальна перевірка системного/текстового повідомлення від банкомата. Скрипт знаходить блок `message` і строго порівнює його текстовий вміст зі строкою (`{string}`) з таблиці (наприклад, звіряє, чи вивелось "Операція успішна" або "Недостатньо коштів").

## 4. Відтворені сценарії

Для зручності та уникнення дублювання коду, в Cucumber використовується **Структура сценарію (Scenario Outline)**:
```gherkin
    Припустимо на моєму рахунку є <баланс> гривень
    Коли я намагаюся зняти <сума> гривень
    Тоді банкомат повинен видати <видана_сума> гривень
    І мій залишок на рахунку повинен бути <залишок> гривень
    І на екрані має з'явитися повідомлення "<повідомлення>"
```
Ми підставляємо туди різні вхідні дані для симуляції різних випадків. Ось перелік усіх 11 розіграних нами сценаріїв (рядків з таблиці Прикладів).

### Сценарій 1: Стандартне зняття готівки
**Опис сценарію:** Користувач знімає суму, що є меншою за його баланс.
**Очікуваний результат:** Відбудеться видача грошей, баланс зменшиться, виведеться повідомлення успіху.
**Код (Дані з таблиці):**
```gherkin
| баланс: 1000 | сума: 500  | очікувано видано: 500 | залишок: 500 | повідомлення: "Операція успішна" |
```

### Сценарій 2: Зняття всіх коштів (під нуль)
**Опис сценарію:** Користувач знімає всю наявну суму на рахунку (баланс після операції дорівнюватиме 0).
**Очікуваний результат:** Система видасть гроші, залишок коректно обнулиться.
**Код (Дані з таблиці):**
```gherkin
| баланс: 500  | сума: 500  | очікувано видано: 500 | залишок: 0   | повідомлення: "Операція успішна" |
```

### Сценарій 3: Маже повне зняття
**Опис сценарію:** З великої суми знімають майже все (залишається рівно 1 гривня).
**Очікуваний результат:** Математична логіка спрацьовує правильно, залишок дорівнює 1.
**Код (Дані з таблиці):**
```gherkin
| баланс: 1000 | сума: 999  | очікувано видано: 999 | залишок: 1   | повідомлення: "Операція успішна" |
```

### Сценарій 4: Дрібна транзакція
**Опис сценарію:** З малого балансу знімається дрібна сума для перевірки звичайних операцій.
**Очікуваний результат:** Гроші успішно зняті.
**Код (Дані з таблиці):**
```gherkin
| баланс: 75   | сума: 25   | очікувано видано: 25  | залишок: 50  | повідомлення: "Операція успішна" |
```

### Сценарій 5: Недостатньо коштів (звичайний кейс)
**Опис сценарію:** Користувач намагається зняти значно більшу суму, ніж є на рахунку.
**Очікуваний результат:** Банкомат не видає гроші, баланс залишається незмінним, система повідомляє про відмову.
**Код (Дані з таблиці):**
```gherkin
| баланс: 200  | сума: 500  | очікувано видано: 0   | залишок: 200 | повідомлення: "Недостатньо коштів" |
```

### Сценарій 6: Спроба зняття з порожнього рахунку
**Опис сценарію:** Баланс нульовий, але користувач пробує щось зняти.
**Очікуваний результат:** Строга відмова і повідомлення про відсутність фінансів.
**Код (Дані з таблиці):**
```gherkin
| баланс: 0    | сума: 100  | очікувано видано: 0   | залишок: 0   | повідомлення: "Недостатньо коштів" |
```

### Сценарій 7: Не вистачає рівно 1 гривні
**Опис сценарію:** Сума запиту перевищує баланс користувача рівно на 1 гривню (межове значення для тестів).
**Очікуваний результат:** Система має жорстко відмовити (0 грн видано), попри мінімальну похибку.
**Код (Дані з таблиці):**
```gherkin
| баланс: 500  | сума: 501  | очікувано видано: 0   | залишок: 500 | повідомлення: "Недостатньо коштів" |
```

### Сценарій 8: Некоректна сума (Нуль)
**Опис сценарію:** Користувач пробує зняти 0 гривень.
**Очікуваний результат:** Система не дозволяє такі безглузді операції, повертає помилку валідації "Некоректна сума".
**Код (Дані з таблиці):**
```gherkin
| баланс: 500  | сума: 0    | очікувано видано: 0   | залишок: 500 | повідомлення: "Некоректна сума" |
```

### Сценарій 9: Некоректна сума (Від'ємна)
**Опис сценарію:** Спроба ввести від'ємне число (-50).
**Очікуваний результат:** Адекватна реакція системи з метою безпеки, заборона зняття та повідомлення про помилку введення.
**Код (Дані з таблиці):**
```gherkin
| баланс: 500  | сума: -50  | очікувано видано: 0   | залишок: 500 | повідомлення: "Некоректна сума" |
```

### Сценарій 10: Спеціально провалений тест (Хак)
**Опис сценарію:** У користувача 100 грн, і він знімає 1000. В коді тесту ми НАВМИСНО вказуємо, що ОЧІКУЄМО успіх. Це зроблено задля того, щоб побачити, що буде, якщо логіка піде не за планом.
**Очікуваний результат (в тесті):** Видасться 1000.
**Фактичний результат (У додатку):** Відмова.
**Результат:** Тест ВПАДЕ з `AssertionError: expected 0 to equal 1000`. Зробиться скріншот екрану.
**Код (Дані з таблиці):**
```gherkin
| баланс: 100  | сума: 1000 | очікувано видано: 1000 | залишок: 0  | повідомлення: "Операція успішна" |
```

### Сценарій 11: Спеціально провалений тест (Помилковий песимізм)
**Опис сценарію:** Баланс (300) цілком покриває запит на 100. Сума коректна. Але в коді тесту ми навмисно налаштовуємо його на очікування відмови.
**Очікуваний результат (в тесті):** Відмова ("Недостатньо коштів").
**Фактичний результат (У додатку):** Гроші видані успішно.
**Результат:** Тест ВПАДЕ через незбіг очікування та реальності (`AssertionError`). Зробиться скріншот успішно проведеної операції.
**Код (Дані з таблиці):**
```gherkin
| баланс: 300  | сума: 100  | очікувано видано: 0   | залишок: 300 | повідомлення: "Недостатньо коштів" |
```
